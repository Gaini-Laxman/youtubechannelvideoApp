You'll also learn the differences between Monolithic and Microservices architectures and how to develop scalable applications.

What is Monolith Architecture?

If we develop all the functionalities in a single project, then it is called a Monolith architecture-based application.

We package the application as a jar/war to deploy it on the server.

Since the monolith application contains all the functionalities, it will become a large jar/war file.

Advantages of Monolith Architecture:

  • Simple to develop
  • Everything is available in one place
  • Configuration is required only once

Disadvantages of Monolith Architecture:

  • Difficult to maintain
  • Dependencies among the functionalities
  • Single point of failure
  • Entire project needs re-deployment

Microservices to Overcome Monolithic Issues:

To overcome the problems of Monolithic architecture, Microservices architecture was introduced.

Microservices is not a programming language, framework, or API. It is an architectural design pattern.

In Microservices, application functionalities are developed with loose coupling. We don't develop all the functionalities in a single project but divide them into several REST APIs. One REST API is called one Microservice.

Advantages of Microservices:

  • Loosely Coupled
  • Easy to maintain
  • Faster development
  • Quick deployment
  • Faster releases
  • Less downtime
  • Technology independence

Disadvantages of Microservices:

  • Bounded context (deciding the number of services)
  • Lots of configurations
  • Visibility
  • Managing many services (Pack of cards)

Microservices Architecture

Microservices architecture does not have a fixed structure. It can be customized based on the project requirements. However, most Microservices projects will use the following components:

  • Service Registry (Eureka Server)
  • Services (REST APIs)
  • Interservice Communication (FeignClient)
  • API Gateway
  • Admin Server
  • Zipkin

Service Registry

Service Registry acts as the database of services available in the project. It provides the details of all registered services. We can use Eureka Server as the service registry, which is provided by Spring Cloud Netflix.

Backend Services

In the project, services are essentially REST APIs. They contain the backend business logic and interact with the database or third-party APIs. For inter-service communication, we can use FeignClient. Multiple instances of a service can run for load balancing.

API Gateway

API Gateway is used to manage the backend APIs. It acts as a mediator between end-users and backend APIs. It contains routing and filter logic (such as authentication) and is registered with the service registry. Spring Cloud Gateway can be used as the API Gateway.

Admin Server

Admin Server helps in managing backend APIs' actuator endpoints. It provides a user interface to monitor these endpoints.

Zipkin Server

Zipkin Server is used for Distributed Tracing. It allows us to monitor which API is taking more time to process requests, helping to identify performance bottlenecks.
